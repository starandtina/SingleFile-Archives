背景介绍

什么是 Chatbot

首先，我们简单的来了解一下 Chatbot。

常见的像苹果的 Siri、Google 的 Google Assistant，天猫，百度的度秘都属于 Chatbot

Chatbot 也就是对话机器人，基于 NLP （自然语言处理）技术，是一种人机交互形式，通过对话的方式来进行交互的

Chatbot 根据呈现包括有交互界面和无交互界面的，像天猫精灵属于无界面，像苹果的 Siri、Google 的 Google Assistant（Google 智能助理）、阿里小蜜属于有界面的。

下面重点介绍一下阿里小蜜。

阿里小蜜 —— 对话即服务

阿里小蜜是基于对话即服务这么一个理念，为用户提供传统意义上的问答、订单咨询等服务，也提供导购、出行等泛服务

目前小蜜覆盖了阿里集团所有生态，包括面向消费者的 50 多个 bu 的业务。

比如像飞猪的行程助理，盒马小蜜，优酷的侯三迷星巴克的智能助理。

架构设计

产品分析

现在大家对阿里小蜜有了一些概念，那接下来给大家分享一下，我们对小蜜产品的分析、对架构的一些思考，以及如何基于业务沉淀出对话领域的UI解决方案。

阿里小蜜的定位是一个“智能客服助理”

  • 技术角度：小蜜是一个同时支持多消息渠道的 WebIM
  • 产品角度：小蜜是一个拥有很多能力的智能机器人，客服能力只是其中一种能力
  • 业务角度：小蜜是一个可以覆盖任何服务场景的智能客服

基于这些视角，小蜜对于我们来说有三个方面的挑战：

  1. 智能的挑战：小蜜可以无限的扩展各种功能
  2. 规模的挑战：我们要做全球最大的服务提供商，所以我们要适配各种端，要满足不同业务的个性化诉求，要协同多个合作团队
  1. 架构的挑战：我们要如何去做好架构分层，同时满足多消息渠道、业务个性化、对话能力增强扽等诉求

基于以上考虑，我们是这么来对架构进行分层设计的：

  • 最底层的适配层，用来解决各种客户端 jsBridge 调用的问题
  • 上面是展示层，用来处理对话界面的展现和交互
  • 和展示层紧密相关的是组件平台，组件平台管理小蜜对话组件，并提供开发接入能力
  • 再上面是消息层，用来处理不同消息渠道 消息的收发、消息的处理
  • 再上面是 context 层，用来串起前后层，目的是为对话组件提供执行环境
  • 最上面是定制层，用来实现小蜜实例的个性化定制功能，包括界面、机器人实例、首页消息等

如何抽象和设计展示层

下面重点介绍一下展示层：Chatbot 领域的展示层和传统的有什么区别，以及我们是如何对展示层做抽象和沉淀的

下面我以小蜜为例子讲一下 chatbot 的交互界面：

用过 IM 产品的都知道用户都是通过最下面的输入框或快捷短语发送消息来进行产品交互的，整个界面的主要内容都集中在中间这块，也就是消息列表，这里有用户发的消息，机器人或者人工回复的消息。其它部分相对固定不变

所以 Chatbot 是可以分为对话容器和交互式卡片两部分

那这样拆解直接发现，展示层主要需要处理三个问题：

展示层要处理的三个问题

  1. 对话容器的抽象
  2. 交互式卡片的开发，如何开发规范统一的卡片模板
  1. 交互式卡片的渲染，如何将卡片渲染在对话容器中

1. 对话容器的抽象

我们对市面上常见的 Chatbot 产品进行比较，比如这里有：

  • 阿里小蜜
  • 招行的我的客服
  • 京东的京东客服
  • Google 的 Google Assistant（Google 智能助理）

通过对比可以发现大家的结构布局都是类似的

  • 都是最上面是导航栏、
  • 然后是消息列表，
  • 然后可能会有快捷短语、
  • 最后是输入区域，
  • 输入区域又能打开工具栏

所以呢，我们把对话容器抽象成了一个对话组件 Chat。它来对整个对话界面进行统一布局，同时也通过它来配置消息渲染函数，对消息进行渲染，对发送消息、点击快捷短语等事件进行绑定。

当然我们对话组件不单单解决了这些问题，包括像刘海屏适配，软键盘处理等一些通用问题也一并解决了。

容器的问题我们通过对话组件已经解决，下面来看一下交互式卡片的开发。

2. 交互式卡片的开发

这块首先我们和设计师一起，对现存的几十张卡片进行走查，对卡片内容进行归纳梳理、调整打磨优化，然后制定出我们的设计规范。

在这过程中，我们也能更深入的了解各业务和各角色的诉求，并且发现哪些内容复用性更高，从中判断应该把哪些组件作为通用组件，然后沉淀下来

有了这些通用组件后，现存的卡片和新开发的卡片就可以直接引用，在统一样式和交互的同时也很大幅度得降低了开发成本。

3. 交互式卡片的渲染

现在已经有了对话组件、有了用通用组件搭建出现的交互式卡片，那交互式卡片是怎么渲染到对话容器中的呢？

我们来看一下这张卡片注册渲染的流程图

首先是卡片的注册，卡片其实本质上是一个 React 组件，根据加载方式的不同我们分成，同步组件和异步组件。

同步组件就是我们放在项目中的组件，异步组件呢是单独打包放到 cdn 上的组件，这样可以按需加载避免影响性能

我们把这两种组件以卡片 code 为索引统一注册到 ComponetsProvider 下面。

当用户发送消息，或者机器人、人工回复消息时，消息通过消息类型或者卡片 code 从 ComponetsProvider 中拿到组件实体，然后渲染到页面上。

当然有些交互式卡片也是有发送消息的能力的，这块其实和用户发送消息是一致的，就不再说了。

这个就是整个卡片注册和渲染的过程。

展示层前面提到的几个问题现在都已经解决了，这块我们抽象后沉淀了一套通用的展示层解决方案 —— 我们叫它 ChatUI

ChatUI 包含了前面提到的对话组件、通用组件、还有其它的一些辅助模块，它基于“主动”、“高效”、“情感”、“沉浸”的设计原则，提供给用户良好的交互体验。还支持主题定制、响应式、国际化、无障碍等特色

通用组件这块，我再简单举个例子：

  • 一个是横向滚动，我们 Chatbot 的界面主要内容都是以消息的形式呈现的，所以不能像其他 h5 页面一样把内容从上往下排列，这样的话其它消息都被一下子刷没了，所以一般都是以横向滚动的形式展示，这块我们开发了一个 ScrollView 组件专门用来解决这类问题；
  • 另一个是富文本内容，富文本内容因为可以插入代码会有安全风险，所以一般都是不建议使用的，但不少时候又不得不使用，特别像 Chatbot 这种场景就会有问答知识需要用富文本来展示，所以这块我们也开发了一个 RichText 组件，它可以把不安全的代码过滤掉，你可以放心的使用富文本。

另外刚提到 ChatUI 的特色，我这边也简单介绍一下。

一个是响应式设计，我们 Chatbot 虽然主要是在移动端使用，但 pc 端也是有不少使用场景的，所以我们也做了响应式设计，响应式设计不是简单的大小的改变，很多交互方式和呈现形式都一并调整了，比如这个例子，当页面变宽后，卡片也相应变大了，但会有个最大宽度不至于一直全宽显示，工具栏也从加号里面移到外面来了用户可以直接点击，之前的上拉弹窗，也变成了居中显示的弹窗

还有个是无障碍支持，这块我们 ChatUI 已经做到了组件级别的支持，使用时并不需要再额外处理，有些需要唯一 id 才能支持无障碍的地方如果用户没有传 ID ChatUI 也会自动生成，像我们这个基于 ChatUI 开发的阿里小蜜就已经通过了信息无障碍研究会的认证。

ChatUI开源

为什么要开源

首先是为什么要开源

第一点 是有庞大的市场规模和需求：根据 IDC 2020 年 6 月发布的《中国人工智能软件及应用市场半年度研究报告》，预计到 2024 年，中国对话式人工智能市场规模将达到 15.6 亿美元，年复合增长率为 30%。但是呢，在前端的对话领域这块，业界的建设和沉淀还是空白的

第二点 是为了领域生态的共建：

因为我们基于自己的业务沉淀了很多卡片、主题，但是放大到整个市场，还是不够的，因此，想借助社区的力量，共建对话领域的生态。另一方面，也可以从大家对 ChatUI 的应用过程中，发现一些创新的场景，反哺我们的业务

基于以上几个原因，最后我们选择了开源。

开源需要考虑的事情

下面是开源需要考虑的事情

文档

第一个是文档，没开源之前，因为都是内部使用，自己就是开发者，不需要文档也知道怎么使用；或者是同事使用，可能会提供个够用的文档就行了，但如果选择了开源，外部的用户对文档这块要求就比较高了，如果没有文档或者文档不好用，他们完全会选择抛弃你选择其它产品，所以有一个完善的文档显得就非常重要。

这块的话，我们 ChatUI 是有一个专门的官网来介绍 ChatUI 和展示文档。

代码质量

第二点是代码质量，在你选择了开源了，就意味的你写的代码会被所有人看到，所以代码光能用、好用已经不够了，你还得把代码写的好看，质量要高，要更加健壮。

这块的话我们 ChatUI 是完全用 TypeScript 开发的，提供完整的类型定义文件，同时用 lint 工具和单元测试来保障代码质量，发布前还会进行 code review，目前 ChatUI 代码覆盖率已经有 90% 左右，还在向 100% 努力。

易用性

再一个是易用性，如果你开源的工具，功能很强大，但是呢非常难用，别人本来是想用开源项目来降低开发成本，用了后才发现入坑了。

像我们 ChatUI 就把很多通用问题解决了，比如之前提到了刘海屏适配、软键盘处理，富文本处理，让用户把更多的时间放到业务逻辑中去

扩展性

我们提供的是一套通用的 chatbot ui 解决方案，默认是我们自己的 ui 样式和交互逻辑，但可能某些业务有自己特殊的需求，我们的通用方案就满足不了了，这块的话，ChatUI 提供很强的扩展性，比如说对话组件，上面的导航栏和下面的输入框，你可以通过传简单的键值对配置来简单配置就能渲染出默认样式的导航栏和输入框，但如果业务有对这块有非常个性化的定制需求，你也可以通过传自定义的渲染函数或者自定义组件来完全覆盖默认样式

如何让开发者开箱即用？

再一个，对于易用性这块，我们把范围扩大到搭建 Chatbot，前面我们已经说过很多 Chatbot 都是类似的，不管是界面还是交互，我们可以让开发者开箱即用吗？

那这就是我们的 ChatUI Pro：

我们在 ChatUI 设计规范和基础组件的基础上，继续向上构建，以阿里小蜜为原型，整合小蜜典型卡片模板和业务组件，加入各业务场景的最佳实践，以配置化的接口的形式封装成一套开箱即用的解决方案 —— ChatUI Pro。

使用 ChatUI Pro 来搭建 Chatbot 页面只需要3步：

  • 第一步是引入 SDK 脚本；
  • 第二步是根据需要进行配置；
  • 第三步是 run，运行起来。

简单 3 步就能搭建出一个 Chatbot

简单不意味的简陋，通过 ChatUI Pro 完备的配置系统，你照样可以配置出很功能强大的 Chatbot，比如，浙江政务服务助理就是用 ChatUI Pro 搭建出来的，可以同时适配移动端和 pc 端。这里额外提一下，在之前疫情开始的时候，用 ChatUI Pro 搭建出来的浙江政务服务助理还被 CCTV 新闻联播报道了。

ChatUI Pro 在我们阿里云小蜜有不少应用，开发上线了十几个 ISV 的智能客服场景，涵盖了政务、金融、运营商、交通等多个行业

未来展望

前面介绍的都是以文本为基础的 Chatbot，但我们并不满足现状，我们还在不断探索和创新，比如尝试 ChatUI 语音版，以全新的界面通过语音进行问答交互，提供更好的人机交互体验；

比如增加 3D 虚拟形象，让 B 端客户拥有自己的品牌形象，可以表达自己的情绪、表情，把交互体验提到新的高度，让用户感受更加情切友好。

相信随着 5G 时代的到来，3D 虚拟形象将会是一种趋势
